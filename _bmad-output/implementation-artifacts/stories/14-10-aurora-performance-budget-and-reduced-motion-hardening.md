# Story 14.10: Aurora performance budget and reduced-motion hardening

## Story

As a **visitor on a low-powered or motion-sensitive device**,
I want **the Aurora glass surfaces to stay lightweight and to honor my reduced-motion preference everywhere**,
so that **the assessment loads fast, scrolls smoothly, and never conveys meaning through motion I cannot see**.

## Acceptance Criteria

1. **Per-surface composite-layer budget asserted by a source guard (PR-29 / design §4.1).** After auditing the Aurora surfaces from Stories 14.3-14.9, no single Aurora rule stacks more than ONE logical `backdrop-filter` layer (a standard `backdrop-filter: blur(var(--surface-glass-blur))` paired with its `-webkit-backdrop-filter` vendor prefix of the SAME blur is one logical layer, not two), and the deep-blur `--glass-blur-lg` (20px, [primitives.css:111](src/css/primitives.css#L111)) is referenced by AT MOST ONE surface — the full-bleed decorative hero glow `.landing__aurora` ([landing.css:42](src/css/components/landing.css#L42)) — never sprinkled across panels. A new source-regex scaffold test (`tests/scaffold/14-10-aurora-perf-reduced-motion.test.mjs`) asserts these caps, splitting each component CSS into rule bodies and counting the unprefixed `backdrop-filter` per rule, and totalling `--glass-blur-lg` references across all Aurora component files (≤1).

2. **Two-layer token architecture preserved + decorative-glow/grid alpha bounded (PR-29 / design §4.1 / UX-DR1).** Components consume only semantic roles: no Aurora component file introduces a literal hex / `rgba()` color (the guard scans the comment-stripped source of every Aurora component file), and every frosted glass blur is driven by a §4.1-capped blur token — the semantic `--surface-glass-blur` role (= `--glass-blur-md`, 12px) for standard panels, or a grandfathered capped `--glass-blur-{sm,md,lg}` radius primitive for the deliberate small chip (`.landing__saved-btn` uses `--glass-blur-sm`, 6px) and the hero glow — never an uncapped literal px. The decorative aurora glow alphas stay ≤0.18 (`--aurora-blue-glow` 0.16, `--aurora-violet-glow` 0.14, and the light analogs) and the grid alpha ≤0.06 (`--aurora-grid-line` 0.05, light 0.045) in [primitives.css](src/css/primitives.css), and the `--backdrop-aurora-1/2` / `--backdrop-grid` semantic roles resolve only to those bounded `--aurora-*` primitives in [semantic.css](src/css/semantic.css) — the cap is centralized in the primitive layer, never bypassed by a brighter literal. Zero third-party runtime deps; no inline `<style>` (NFR6/NFR7 CSP).

3. **Per-scene reduced-motion static end-state on every entrance/decorative-animation surface (PR-29 / design §4.2).** The aurora is STATIC by default — no continuous, looping, or parallax motion anywhere (the guard forbids `animation-iteration-count: infinite` and the `infinite` keyword in any `animation:` shorthand across every Aurora component file, a §4.2 hard prohibition / §6 anti-pattern 3). Every Aurora surface that carries ENTRANCE / DECORATIVE animation (a `@keyframes` + `animation:` with a *deliberate final frame*) ships its OWN per-scene `@media (prefers-reduced-motion: reduce)` block that pins the static end-state (`animation: none` + the final `opacity`/`transform`), mirroring `.landing` / `.landing__aurora` at [landing.css:182](src/css/components/landing.css#L182) — so the reduced-motion composite is the deliberate end-state, not a frozen mid-animation frame, rather than relying only on the global duration-zeroing net at [base.css:163](src/css/base.css#L163). AUDIT outcome: `landing.css` is the only Aurora component carrying `@keyframes`/`animation:`, and it ALREADY ships that per-scene rule; the remaining Aurora surfaces carry only interaction transitions (hover/focus color/border/box-shadow), whose resting (unhovered) state is already the correct static end-state and is covered by the global net per design §4.2 — so no per-scene reduce rule is *genuinely missing*. The guard additionally pins that the two surfaces that already set the defense-in-depth precedent (`landing.css`, `selection-scene.css`) keep their per-scene reduce fallback. No information is conveyed by motion alone (WCAG 2.2 AA, Epic 13 §4.3 contract).

4. **Progressive mobile simplification (PR-29 / design §4.1).** Below `--bp-tablet` (600px) the Aurora surfaces simplify and no oversized full-bleed `--glass-blur-lg` blur appears on phones. AUDIT outcome: the only `--glass-blur-lg` consumer is the decorative `.landing__aurora` glow (a `filter:`, not a glass surface's `backdrop-filter`), which the global reduced-motion net + its per-scene reduce rule already neutralize, and which carries no text; the glass *surfaces* all use the bounded `--surface-glass-blur` (12px) or the 6px chip blur. The existing mobile step-downs at [base.css:111](src/css/base.css#L111) (heading sizes), [landing.css:195](src/css/components/landing.css#L195), [selection-scene.css:140](src/css/components/selection-scene.css#L140), [item-runner.css:345](src/css/components/item-runner.css#L345), and [score-panel.css:240](src/css/components/score-panel.css#L240) already step the heaviest surfaces down; no surface was found that genuinely lacks a needed step-down, so none was added. The frozen Epic 11/13 DOM contracts (`section.landing`, `h1#landing-heading`, `#start-test-btn`, `.landing__methodology-link`) are preserved — styling only.

5. **Opaque-fill degradation fallback on every glass surface (PR-29 / design §6 anti-pattern 6).** Each Aurora glass surface keeps its `@supports not (backdrop-filter: blur(1px))` fallback that drops to the strong opaque fill (`--surface-glass-strong`), so the AA contrast guarantee is carried by the fill alpha, never by the blur, on every degradation path. The guard asserts the fallback exists and drops to `--surface-glass-strong` on each of the ten glass-surface component files (`landing`, `aurora`, `glass-surface`, `masthead`, `consent-scene`, `chrome-header`, `chrome-footer`, `item-runner`, `score-panel`, `saved-results`). The co-equal Percentile / IQ-scale / Range triplet members (`.score-panel__percentile` / `.score-panel__anchor` / `.score-panel__band`) remain styled and legible (the guard asserts they are still present), preserved across every degradation path.

6. **Lightweight path does not re-introduce the imperceptible-glass regression; rendered perf/visual verification deferred to 14.11; build byte-stable.** The bounded-blur lightweight path still reads as depth against the deep-navy spatial backdrop (the glass fill RGB stays measurably different from the backdrop, per design §6 anti-pattern 1) — this is the rendered guarantee owned by Story 14.11's committed visual-regression baselines on `ubuntu-latest`, NOT asserted in source. The Lighthouse FCP/LCP perf budgets and the light/dark visual baselines are the dormant Playwright/perf legs whose baselines 14.11 commits; this story does NOT activate the visual-regression CI job in [pr-checks.yml](.github/workflows/pr-checks.yml) (that is 14.11's job). Because the audit found the codebase already compliant on every §4.1/§4.2/§6 budget, this story ships the source guard + this spec and modifies NO component CSS — the build stays byte-stable (NFR21) and `css-components-lines` stays unchanged at 2336/2600. No EN string is touched, so there is no NFR27 PL/RU cascade and no `sourceHashEN` re-bump.

## Dev Notes

### Implementation strategy
- **This story is HARDENING, not a rewrite.** The audit (Tasks 1-3) found the Aurora surfaces from Stories 14.3-14.9 ALREADY compliant on every design §4.1/§4.2/§6 budget. So the deliverable is the source guard that LOCKS those budgets in + this spec, with NO component CSS modified — a valid guard-only outcome that keeps the build byte-stable (NFR21).
- **Composite budget — already within caps.** Every glass surface declares exactly one `backdrop-filter: blur(var(--surface-glass-blur))` paired with its `-webkit-` prefix (the same logical layer, not two stacked filters); the page backdrop is PAINTED (`background-image` gradients on `body`, no filter — design §2.1), so it costs no composite layer. `--glass-blur-lg` (20px) is referenced by exactly one surface — the decorative `.landing__aurora` full-bleed hero glow — exactly as primitives.css:111 reserves it. The glow/grid alphas are bounded in the primitive layer (0.16/0.14 ≤0.18; 0.05 ≤0.06), and the `--backdrop-*` roles resolve only to those primitives, so the cap cannot be bypassed by a component.
- **Reduced-motion — the model already covers the only animated scene.** Among the Aurora component files, `landing.css` is the ONLY one with `@keyframes`/`animation:` (the hero `landing-enter` fade-rise + the `landing-aurora-drift` decorative entrance), and it already ships the per-scene `@media (prefers-reduced-motion: reduce)` rule pinning `animation: none` + the deliberate final `opacity`/`transform` — the MODEL the AC names. The other surfaces carry only interaction transitions (hover/focus color/border/box-shadow at 120ms); their resting state IS the correct static end-state, and the global duration-zeroing net (base.css:163) is the documented mechanism for those per design §4.2 — adding a per-scene `transition:none` to each would be redundant defense-in-depth, NOT a genuine gap, and would risk scope-creep into non-Aurora chrome. The guard therefore asserts a per-scene static end-state for *animation-bearing* scenes (where a frozen mid-frame is a defect) and pins the existing precedent reduce rules (`landing`, `selection-scene`), rather than forcing a redundant rule onto every transition.
- **The "no continuous motion" hard prohibition is now guarded.** Design §4.2 / §6 anti-pattern 3 bans looping/parallax motion; the guard asserts no `animation-iteration-count: infinite` and no `infinite` keyword in any `animation:` shorthand across every Aurora component file — so a future story cannot add a drifting/looping aurora.
- **Mobile + degradation — already present.** No `--glass-blur-lg` reaches a glass surface on phones (its only consumer is the decorative glow `filter:`, which carries no text and is neutralized by reduced-motion); the heaviest surfaces already step down at `@media (max-width: 30rem)`. Every one of the ten glass-surface components keeps its `@supports not (backdrop-filter: blur(1px))` fallback dropping to `--surface-glass-strong` (the AA contrast guarantee is the fill alpha, never the blur). The co-equal triplet stays styled across every path.
- **Grandfathered blur-radius primitives are a COUNT check, not a ban.** The two-layer rule forbids components from consuming color/contrast primitives (`--glass-*` fills, `--color-neutral-*`) and literal hex; it does NOT forbid the capped `--glass-blur-{sm,md,lg}` RADIUS primitives, which a couple of component files (e.g. `.landing__saved-btn` → `--glass-blur-sm`) already consume. The guard's blur assertion allows the semantic `--surface-glass-blur` role OR a capped `--glass-blur-*` primitive and only forbids an uncapped literal px; its `--glass-blur-lg` assertion is about COUNT (≤1 surface), not forbidding the ref.
- **No JS / markup / i18n / corpus / pr-checks.yml touched.** The change is the new node guard + this spec only. The visual-regression CI job in pr-checks.yml stays dormant — Story 14.11 flips it. No EN string is touched → no NFR27 PL/RU cascade and no `sourceHashEN` re-bump. The frozen Epic 11/13 DOM contracts and the co-equal triplet are preserved (restyle-only story, and here even the styling is unchanged).
- **Budget.** No `src/css/components/**` file is modified, so `css-components-lines` stays at 2336/2600. The guard lives under `tests/scaffold/**`, outside the components budget domain.
