# Story 8.3: scheduled.yml workflow — mirror parity + archival health + failure routing

## Story

As a **maintainer running a solo project with no email/Slack integrations (NFR6)**,
I want **the `scheduled.yml` workflow (stub from Epic 1) fully implemented with weekly mirror-parity + archival-snapshot health checks that route failures to labeled GitHub Issues (per Architecture's John gap #6)**,
so that **silent drift is caught (canonical and mirror diverging, IA snapshot 404ing, Zenodo DOI not resolving) and the maintainer's weekly triage has a single dashboard — no third-party monitoring SaaS, no paging**.

## Acceptance Criteria

> **Epic-8 dev-phase decision (infra-now) — no live cron, no live HTTP probes, no `gh` API.** Same posture as Story 8.1 / Story 8.2 / Story 7.8. `scheduled.yml`'s weekly cron only fires against the live repo at launch (Epic 10) when GitHub Pages + the mirror (Story 8.4) + the recorded archival URLs (populated by the first `corpus-v*` release) exist. The dev-phase deliverable is: (a) the **fully-authored workflow** (4 health-check jobs, each carrying a per-job `area:<check>` label, + a failure-routing job/step that opens-or-appends a labeled GitHub Issue); (b) `docs/scheduled-yml-failure-routing.md` **finalized** (it does NOT exist yet — there is no Epic-1 stub on disk, only an epics.md reference); (c) a **structural contract test**. NO cron run, NO live `curl`/HEAD probe, NO `gh issue create` runs in dev — every network/API touch sits behind a launch gate (a repo var like the 8.2 `vars.IQME_LIVE_ARCHIVAL` pattern, OR cron-only steps that no-op without the gate). The structural contract is verified by tests parsing `scheduled.yml` + asserting the doc exists.
>
> **Scope boundary.** This story REPLACES the `scheduled.yml` Epic-1 stub (the single `scheduled-check` job whose only step is `echo "Activates in Epic 8"`) with the real jobs. It does NOT deploy the Story 8.4 mirror — the `mirror-parity-check` job is **self-gating** so 8.3 lands cleanly BEFORE 8.4 (Winston's intra-epic sequencing concern). The full `CONTRIBUTING.md` cross-link to `docs/scheduled-yml-failure-routing.md` is **Story 8.6's** responsibility (CONTRIBUTING.md is rewritten there) — 8.3 finalizes the doc and may add a minimal one-line stub reference in CONTRIBUTING's existing "Expanded in Epic 8" section, but MUST NOT block on the full rewrite.

1. **AC-1 (weekly cron + 4 health-check jobs, each `area:<check>` labeled):** `.github/workflows/scheduled.yml` triggers `on: schedule:` with a **weekly** cron (e.g. `cron: "0 6 * * 1"` — Monday 06:00 UTC; the Epic-1 stub's daily `0 6 * * *` is changed to weekly per the epics AC "runs weekly via cron"). It defines four jobs: `mirror-parity-check`, `internet-archive-snapshot-health`, `software-heritage-snapshot-health`, `zenodo-doi-resolution`. Each job carries a **per-job `area:<check>` label** — declared so it is grep-able in the workflow text (the label the routed Issue will also carry: `area:mirror-health`, `area:archive-health`, `area:archive-health` / a SH-specific token, `area:doi-health` respectively — see Dev Notes for the exact label vocabulary aligned with Story 8.2). The `echo "Activates in Epic 8"` stub job is gone.
2. **AC-2 (job semantics — response-body parity + HEAD health, dev-inert):** `mirror-parity-check` compares the **response body** of the canonical GitHub Pages URL vs the Codeberg/Cloudflare mirror URL for key pages (**headers exempted** per architecture — only the body is compared). `internet-archive-snapshot-health` HEAD-requests the IA snapshot URLs recorded in `docs/launch-readiness/internet-archive-snapshots.md` and asserts HTTP 200. `software-heritage-snapshot-health` HEAD-requests the SH save/archived-origin URLs recorded in `docs/launch-readiness/software-heritage-snapshots.md`. `zenodo-doi-resolution` HEAD-requests the recorded DOI(s) (read from the canonical sink `CITATION.cff` `doi:`) and asserts resolution to a real Zenodo record. **In dev none of these probes runs live** — every HEAD/`curl` is gated behind the launch path (the live-probe repo var / cron-only no-op); the dev artifact is the structurally-correct jobs naming the real endpoints + reading the real record files, NOT live probe results (no fabricated 200s — lesson-2026-06-04-002).
3. **AC-3 (self-gating mirror-parity — 8.3 lands before 8.4):** the `mirror-parity-check` job is **self-gating**: it first HEAD-requests the mirror URL, and if the response is **non-200** (e.g. the Story-8.4 mirror is not yet deployed), it **skips** the parity comparison and logs a `"mirror not reachable — first deploy pending"` notice rather than **failing**. So Story 8.3 can land before Story 8.4 without producing spurious mirror-parity failures. The self-gating HEAD-then-skip logic is structurally present and grep-able (the skip-notice string + the non-200 guard), inert in dev like the other probes.
4. **AC-4 (failure routing → labeled GitHub Issue, open-or-append dedup):** when any check job fails, a failure handler (a dedicated routing job that `needs:` the check jobs, OR a per-job `if: failure()` step — engineer's call, documented) opens a GitHub Issue labeled **`area:scheduled-check`** PLUS the per-check specific label (`area:mirror-health` / `area:archive-health` / `area:doi-health` per the failing check), using `gh` CLI or `actions/github-script`. The Issue body carries the **failure context**, the **broken URL(s)**, and a **link to the action run** (`$GITHUB_SERVER_URL/$GITHUB_REPOSITORY/actions/runs/$GITHUB_RUN_ID`). **Dedup:** if an Issue with the same label set is **already open**, the handler **appends a comment** rather than opening a duplicate (the open-or-append pattern is structurally present and grep-able). In dev the live `gh` API call does not run (gated behind the launch path).
5. **AC-5 (`docs/scheduled-yml-failure-routing.md` finalized — NEW, no Epic-1 stub on disk):** `docs/scheduled-yml-failure-routing.md` is **created/finalized** (it does NOT currently exist — `ls docs/` confirms; the epics.md "Epic 1 stub" was never materialized) in the project's plain-language docs style. It explains: the **weekly cadence**; the **per-check labels** (`area:scheduled-check` + the four specifics); the **triage discipline** ("maintainer triages weekly; no email/Slack notification" — NFR6/NFR8/NFR35); the **per-check mitigation playbooks** (mirror-divergence → assess + possibly re-deploy/failover; IA 404 → re-snapshot; SH missing → re-save; DOI not resolving → check Zenodo record); the **mirror-parity response-body-only check** (headers exempted, and the self-gating "mirror not reachable — first deploy pending" behavior for the pre-8.4 window). No fabricated URLs or fabricated Issue numbers.
6. **AC-6 (CONTRIBUTING.md cross-link is Story 8.6's job — minimal stub only here):** the epics AC says `docs/scheduled-yml-failure-routing.md` "is referenced from `CONTRIBUTING.md`". The **full** `CONTRIBUTING.md` (which institutionalizes the weekly-triage discipline in the contributor flow) is **Story 8.6**. This story MUST NOT block on the 8.6 rewrite: it may add a single minimal reference line to the doc from CONTRIBUTING.md's existing "Expanded in Epic 8" section (or note the cross-link as 8.6's responsibility) — the load-bearing cross-link is explicitly **8.6's** deliverable, not 8.3's. (If CONTRIBUTING.md is touched, no existing lint must go red — `CONTRIBUTING.md` is not currently `lint-reading-level`-gated; the full plain-language rewrite is 8.6.)
7. **AC-7 (frozen class-A test graduation + new structural test + regression):** the **test-author phase** (NOT this spec's job, but a required, explicit task — see Tasks/Subtasks) must: (a) **graduate** the `tests/scaffold/ci-matrix.test.mjs` (class-A) **AC-6 scheduled.yml** test from **stub → activated** (exactly as Story 8.1 graduated the **release.yml** half of AC-6) — the assertion that `scheduled.yml` STILL contains `echo "Activates in Epic 8"` flips to asserting the **activated** structure (stub placeholder gone; the four check jobs present; the failure→labeled-Issue routing still present, now as a real handler rather than an inline-comment); (b) add a NEW `tests/scaffold/scheduled-workflow.test.mjs` (structural: the four jobs declared, the per-job `area:` labels, the failure-routing labeled Issue + dedup-append, the self-gating mirror check + its skip-notice string, `docs/scheduled-yml-failure-routing.md` exists); (c) **re-register integrity** for the edited class-A `ci-matrix.test.mjs` via `tds integrity record` and re-grep `state-manifest.yaml` after the next sweep (lesson-2026-05-20-007 / lesson-2026-05-19-013). `make test` green, `make lint` green, `make build` deterministic.

- **Label vocabulary — align with Story 8.2 + the epics AC.** The epics AC-2 names the routed-Issue labels: `area:scheduled-check` (always) + a per-check specific (`area:mirror-health`, `area:archive-health`, `area:doi-health`). Story 8.2's `release.yml` already routes archival-time failures to a `archival-health`-labeled Issue (the SAME drift surface). DECISION for the engineer: use the **epics-AC `area:*` vocabulary** (`area:scheduled-check` + `area:mirror-health`/`area:archive-health`/`area:doi-health`) as the canonical labels; both the IA-health and SH-health checks may share `area:archive-health` (the epics give only three specifics for four checks — SH and IA are both "archive health"; if you prefer a distinct SH token, add `area:archive-health` + a SH discriminator in the title, but keep `area:archive-health` so 8.6's `docs/required-ci-checks.md` enumeration and the 8.2 archival-health surface line up).
