# Story bridge-7-8-2-tighten-unfreeze-windows-yaml: Tighten unfreeze-windows.yaml schema lock

## Story

Pre-commit hook (bridge-6-7-3 Phase B) parses unfreeze-windows.yaml via bash+awk on the canonical single-line scalar form (ADR-0014). If a future story writes a multi-line scalar or flow-style entry, the hook silently exits 0 and treats it as no-window — false-positive warning on class-A edits that ARE under an active unfreeze. Either add a sibling schema-validator that runs alongside writes (preferred — same code path as state-manifest's Class I lock) or amend ADR-0014 to enumerate unfreeze-windows.yaml explicitly in the canonical-form requirement. Low coupling; one ADR amendment OR one ~50-line validator. Time-box 1 day.

## Specialist Self-Review

**Decisions made:**
- **Task 1 — validator over ADR-only** (per spec default + user confirmation): added a write-time `assertCanonicalWindows()` in the sibling module rather than amending ADR-0014 only. Runtime enforcement gives the same fail-fast guarantee as the state-manifest Class-I write-lock, so an off-canonical entry can never reach disk where the ADR-only path would rely on convention.
- **Validator scope = single-line-scalar guarantee.** Each window field must be a primitive (string/number/boolean) with no CR/LF; this is exactly what `stringifyYaml(…, { lineWidth: 0 })` renders as one `<key>: <value>` line and what the bash+awk parser can pair. Newline / nested / flow-style → `throw UnfreezeWindowsSchemaError`.
- Hooked the validator into `writeWindows()` (the single choke-point all of openWindows/closeWindow funnel through) rather than each public mutator — one guard, no path can bypass it.

**Alternatives considered:**
- ADR-0014 amendment only — rejected per Task 1 default; no runtime teeth.
- Enforcing the awk parser's exact `file`-first/`expires_at` field layout — rejected: the production writer emits `story_id`-first/`opened_at` (no `expires_at`), so a layout-strict validator would reject every real window and break `tds story unfreeze-tests`. See uncertainty below.

**Framework gotchas avoided:**
- `yaml` lib with `lineWidth: 0` already prevents long-line folding, so the realistic non-canonical vector is an embedded newline (→ block scalar) or a non-scalar value — the validator targets exactly those, not line length.
- Sibling pre-commit auto-bumps patch (v6.5.40 → v6.5.41) and rebuilds the bundle; the bundle sha256 + module.yaml expected_payload_sha256 churn rides in the same commit (expected module behavior, not hand-edited).

**Areas of uncertainty (auditor: please weigh):**
- **Latent field-layout mismatch, deferred.** The pre-commit hook's awk parser (init.ts) scans `^  - file:` + `^    expires_at:` and treats `expires_at > now` as "active". But the production writer (`UnfreezeWindow`) emits `story_id`-first and has **no `expires_at`** field (windows are closed by integrity-record, not expiry). So a *real* `tds story unfreeze-tests` window would not be detected by the hook → the very false-positive this story targets persists for production windows. The existing precommit-hook test passes only because it hand-writes the awk-compatible `{file, story_id, expires_at}` shape, masking the mismatch. This is **deeper than the single-line lock** this story scopes (AC1 conflates "single-line" with the specific field layout); fixing it means reconciling the writer schema with the awk parser (add/compute `expires_at`, reorder, or rewrite the parser) — a design change beyond the 1-day time-box. Flagged here as a candidate follow-up bridge finding rather than silently scope-crept.

**Tested edge cases:**
- accept canonical single-line entry; accept empty schema (`windows: []`, no false-negative — AC4); reject newline-bearing field (AC2); reject non-scalar/array field (flow-style risk, AC2); `openWindows` rejects + persists nothing; single-line round-trip line-shape assertion (AC1).
- Regression: full sibling suite green (97 files / 1385 tests), incl. the existing precommit-hook suppression test (AC3a) and the bridge-1-2-1 unfreeze-tests CLI test (real-window writes still pass the new guard).

## Auditor Findings (round-1)

### [warn] The shipped single-line schema lock (assertCanonicalWindows) is sound, but a deeper latent field-layout mismatch remains: the production UnfreezeWindow writer emits story_id-first / opened_at (no expires_at), while the pre-commit hook's awk parser scans file-first / expires_at and treats expires_at > now as "active". So a real `tds story unfreeze-tests` window is still NOT detected by the hook, meaning the original false-positive-warning failure mode persists for production windows even after this story. The existing precommit-hook test passes only because it hand-writes the awk-compatible {file, story_id, expires_at} shape, masking the mismatch. This is deeper than the 1-day single-line-lock scope and was explicitly deferred (not silently dropped).

- **Category:** latent-defect-deferred
- **Suggested bridge:** `Reconcile the UnfreezeWindow writer schema with the hook's awk parser — add/compute an expires_at field and align field order, OR rewrite the awk parser to match the writer's story_id/opened_at layout — and add a regression test using a real writer-produced window (not a hand-crafted shape).`
