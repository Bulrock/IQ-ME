## What went well

- **Architecture D3 contract held end-to-end.** Five stories (2-1, 2-2, 2-4, 2-5, 2-7) faced an explicit D3-canonical vs epic-narrative naming choice (`quadraturePoints` vs `gridPoints`, `eapEstimate` vs `estimateTheta`, `standardError` vs `posteriorSE`, `scoreSession({...})` vs `scoreSession(positional)`, `claim-id` digit-string vs decimal-dot). Every story made the same call: D3 wins as contract; epic-narrative names ship as zero-cost re-export aliases (`export { X as epicName }` or single-line positional shim). No math duplication anywhere. Pattern captured as [L4](#lessons-captured).
- **R-mirt parity: bit-perfect MATCH on both smoke (6 entries) and full (1000 entries) sets.** Max θ drift 0.000000, max SE drift 0.000000 at 6-decimal precision. The audit chain `v0.1 placeholder → v0.2 JS regen → v1.0 R-validated (smoke) → v2.0 JS-derived (full) → v2.1 R-validated (full)` documented in `tests/golden/CHANGELOG.md`. Third-party reproducibility anchored by SHA256 pins (`vectors.json` a8b2653a..., `r-output-smoke.json` ff62bd5d...).
- **NFR33 (zero runtime deps) held under stress.** Pure-JS Abramowitz-Stegun 7.1.26 standard-normal CDF (~7.5e-8 abs error), hand-rolled log-sum-exp (`log1pExp` helper), M-shift posterior stabilization for 16-item underflow, Mulberry32 PRNG (~10 LOC), regex-based YAML frontmatter parser. No `package.json` introduced; lint-claims-manifest tool is ~50 lines manual regex validation, not AJV.
- **Bridge-1-2-1 `tds story unfreeze-tests` CLI proved out in steady-state usage.** Used correctly 3× in epic-2: story-2-5 post-audit fixture regen (`vectors-smoke.json` + `index.test.mjs:AC-7.4`), story-2-6b cross-story `ci-matrix.test.mjs` activation (`EPIC_1_ACTIVE += golden-vector-parity`), story-2-7 `schema.test.mjs` claim-id refresh. All three followed: `unfreeze --reason=...` → `Edit` → `integrity record --notes=...`. The friction-hostile slip pattern from epic-1 (lesson-2026-05-19-001) is now mechanically prevented.
- **Red-phase discipline survived 4 stories.** The Story 2-1 parity test stayed red through stories 2-2/2-3/2-4 close (failed at progressively deeper points in the call chain: `scoreSession` stub → `eapEstimate` stub → `standardError` stub → finally green at story-2-5). No specialist short-circuited by stubbing `scoreSession` to return mock values to "let tests pass earlier".
- **Forbidden-quadrant matrix 0 violations.** 370 events recorded across roles (test-author, engineer, auditor, writer); every op `allow` or `allow-conditional`. The role-permission boundary held even in cross-story refresh cases (engineer integrity-record after test-author handoff; auditor integrity-ops after engineer commit).
- **Hotfix PR #5 followed deliver-then-fix-forward cleanly.** First R-CI dispatch failed at the DoF error (regenerate.R's `mirt()` initial fit triggered EM with nItems=1 in smoke set). One-line fix (`mirt(..., pars="values")`) landed via PR #5 without epic rollback; v1.0 + v2.1 CHANGELOG entries authored post-validation. The post-merge audit gate worked exactly as designed.

## What broke / what's friction-hostile

- **First R-CI dispatch failed at runtime — the validation path wasn't exercisable locally.** `regenerate.R` was hand-validated against mirt docs but the initial-parameter-template fit code path wasn't run locally (no R installed). nItems=1 smoke entries (2/3/5) tripped DoF=1 < 2-free-2PL-params during the EM fit that was inadvertently triggered. Predictable in hindsight: every story with "Areas of uncertainty" mentioning "first CI dispatch outcome" or "first dispatch will validate end-to-end" predicts a follow-up hotfix PR. Same shape recurred from epic-1 (CI matrix activation paths). **See [L1](#lessons-captured).**
- **Self-circular parity claim posture for ~24h.** Stories 2-5, 2-6a, 2-6b each shipped with `expected` values generated by the SUT impl itself. Auditor caught all three as `[info] deferred-validation` findings; the two-file design + CHANGELOG provenance preserved the audit signal; outcome resolved cleanly with bit-perfect R-mirt MATCH. The in-repo posture during that window read "third-party validated" via parity tests but was actually "self-consistent, NOT third-party validated" until first CI dispatch — only readable by following CHANGELOG v0.2 to v1.0/v2.1. **See [L3](#lessons-captured).**
- **`claim-id` schema-vs-narrative collision (story-2-7).** Epic-narrative authored `golden-vector-parity-0.001-logits`; frozen `METHODOLOGY_CLAIMS.schema.json` regex `^[a-z][a-z0-9-]*$` (story-1-4 class-A deliverable) forbids dots. Specialist transformed to `golden-vector-parity-0001-logits` and documented the deviation. Settled decision for THIS claim, but epic-narrative still reads the dotted form — naming drift between docs. Recurrence risk: future claims like `norming-iq-100-15` would face the same collision if narrative wrote `iq-100.0-15.0`. Second such regex-vs-prose collision in the project (epic-1's lesson-2026-05-19-002 was the first — frozen-test regexes forced unnatural prose workarounds). **See [L2](#lessons-captured).**
- **Cross-repo integrity drift carried over from bridge-1-2-1.** `tds integrity verify` reports 1 failed entry: `../bmad-tds-module/src/cli/__tests__/bridge-1-2-1-unfreeze-tests.test.ts` (sibling-repo path registered during the bridge that spanned IQ-ME + bmad-tds-module). All 50 in-repo IQ-ME integrity entries pass; the failed entry is pre-existing and not introduced by epic-2. Honestly disclosed in story-2-1 self-review #2 + auditor finding [info] integrity-cross-repo. Action: low-priority TDS hygiene; user opted to handle inline whenever a future preflight gate trips on it rather than land a dedicated bridge story.
- **`scoreSession` builds quadrature twice per call.** Story-2-5 honors D3 signature `standardError(theta, responses, items)` (no `quad` param), which forces `standardError` to rebuild a 61-point quad internally; `scoreSession` also builds quad directly. Net ~20μs per call. Acceptable for v1 request volume; story-2-4 + story-2-5 both documented the tradeoff. Optimization deferred to whenever it matters (likely never).

## Lessons captured

- [lesson-2026-05-19-009](../../_tds/memory/lessons.yaml) — **L1 (process, medium):** Deferred-validation gates requiring external runtime routinely surface post-merge as hotfix PRs because the workflow can't be exercised locally; mitigate via local-install OR pre-merge `workflow_dispatch` from feature branch OR pre-enumerated failure-mode hotfix bridge in next epic's backlog.
- [lesson-2026-05-19-010](../../_tds/memory/lessons.yaml) — **L2 (tooling, medium):** Epic-narrative authoring proposes identifiers (claim-id, slug, key) without pre-validating against frozen schema regex; specialists transform mid-impl, creating naming drift between docs. Mitigate via grep + schema-regex validation in pre-story-creation step.
- [lesson-2026-05-19-011](../../_tds/memory/lessons.yaml) — **L3 (process, low):** "Self-circular by construction" parity is a legitimate intermediate state but the in-repo posture must carry explicit provenance (test-file header + CHANGELOG entry naming the deferred validator + outcome placeholder) — not rely on the auditor catching it.
- [lesson-2026-05-19-012](../../_tds/memory/lessons.yaml) — **L4 (process, low):** Architecture D3 canonical names win over epic-narrative descriptive names in cross-domain interfaces; epic-narrative names ship as zero-cost re-export aliases (not duplicate impls). Pin the decision upfront in story Dev Notes; do not relitigate per story.
