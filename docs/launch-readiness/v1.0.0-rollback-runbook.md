# IQ-ME v1.0.0 Emergency Rollback Runbook

> **PURPOSE:** Pre-validated emergency rollback procedure for v1.0.0 launch.
> Read this BEFORE pushing the v1.0.0 tags.
>
> **When to use:** A critical bug discovered within the 48-hour post-launch
> monitoring window (Story 10.3) that cannot be hotfixed quickly. For
> non-critical issues, prefer a v1.0.1 patch rather than tag rollback.
>
> **Story 10.3 dry-run:** This runbook must be walked through as a dry-run in
> Story 10.3 (every step executed except the actual push/withdraw calls) to
> confirm each command is executable before it is ever needed in anger.

---

## When rollback is appropriate

Use this runbook only when **all three** are true:

1. A defect is discovered within the 48-hour post-launch monitoring window
2. The defect cannot be hotfixed with a v1.0.1 patch within ~2 hours
3. The defect materially harms users (broken scoring, broken consent flow,
   wrong clinical-register copy served, privacy breach)

For issues discovered after 48 hours → v1.0.1 release process. Do not rollback
after the window; the Zenodo DOI will already be cited in the wild.

---

## Step 1 — Tag deletion (emergency rollback only)

> ⚠️ **Exception to the no-force-push rule.** Tag deletion with
> `git push origin --delete` is the ONLY authorized destructive git operation
> in this project. It is allowed exclusively for emergency launch rollback.

### 1a — Delete the local tags

```sh
git tag -d app-v1.0.0
git tag -d corpus-v1.0.0
```

**Dry-run verification:** Run `git tag -l 'app-v1.0.0' 'corpus-v1.0.0'` — must return empty after deletion.

### 1b — Delete the remote tags

```sh
git push origin --delete app-v1.0.0
git push origin --delete corpus-v1.0.0
```

**Dry-run verification:** Run `gh api repos/{owner}/{repo}/git/refs/tags/app-v1.0.0 2>&1` — must return 404 after deletion.

> **Note on GitHub Actions:** Deleting the tags will NOT re-trigger
> `release.yml` (it only fires on tag push, not tag delete). The GitHub Pages
> deployment is unaffected by tag deletion — the gh-pages branch keeps its
> content until a new deployment overwrites it.

## Step 2 — Redeploy prior known-good version

After deleting the tags, the live GitHub Pages deployment still shows the
v1.0.0 artifact (tag deletion does not revert gh-pages). To serve a prior
version:

### 2a — Identify the known-good commit

```sh
git log --oneline --decorate main | head -20
```

Find the commit immediately before the v1.0.0 merge (or the last known-green
state). Note the commit SHA.

**Dry-run verification:** Run this command; confirm output is readable and the
pre-v1.0.0 commit is identifiable.

### 2b — Build and deploy the known-good artifact

```sh
git checkout <known-good-sha>
make build
```

Then trigger a manual GitHub Pages deployment by pushing the `dist/` output to
the `gh-pages` branch:

```sh
# Using the same peaceiris/actions-gh-pages pattern from release.yml:
# Option A — trigger via a manual workflow_dispatch if release.yml supports it
gh workflow run release.yml --ref <known-good-sha>

# Option B — direct gh-pages branch push (bypass CI; emergency only)
git subtree push --prefix dist origin gh-pages
```

**Dry-run verification:** Confirm `make build` exits 0 on the known-good commit.
Do NOT push to gh-pages during dry-run.

### 2c — Verify the rollback deployment

After deployment:

```sh
curl -fsSL https://iq-me.org/ | grep '<title>'
```

Confirm the served content matches the known-good version.

## Step 3 — Zenodo DOI withdrawal

Zenodo DOIs are minted per-release. If v1.0.0 is withdrawn, the DOI should
be marked accordingly.

### 3a — Log into Zenodo

Navigate to `https://zenodo.org/` and log in with the project account.

### 3b — Locate the v1.0.0 record

Find the deposit for `IQ-ME v1.0.0` in the Zenodo dashboard.

### 3c — Mark as withdrawal

Zenodo permits marking a record as withdrawn with a public notice:

1. Open the record → Edit → "Retire record / Withdrawal"
2. Enter a brief public withdrawal reason (e.g. "Critical bug discovered at
   launch; withdrawn pending v1.0.1 fix. See GitHub repository for status.")
3. Submit. The DOI will resolve to a withdrawal notice rather than the deposit.

> **Important:** Zenodo withdrawal is permanent for that record. The DOI will
> continue to resolve (to a withdrawal notice) — it cannot be fully deleted.
> When v1.0.1 ships, a new DOI will be minted. Update `CITATION.cff` with the
> new DOI.

**Dry-run verification:** Navigate to the Zenodo dashboard and confirm the
v1.0.0 record is accessible. Do NOT withdraw during dry-run.

## Step 4 — Internet Archive correction notice

The `corpus-release` job in `release.yml` snapshots corpus pages to the
Internet Archive via Save Page Now. If v1.0.0 is rolled back, the IA snapshots
remain (IA does not delete snapshots). Issue a correction notice:

### 4a — Save Page Now on the corrected version

Once the known-good version is deployed to the canonical URL, re-run Save Page
Now to create a newer snapshot that supersedes the v1.0.0 snapshot:

```sh
curl -fsSL "https://web.archive.org/save/https://iq-me.org/" > /dev/null
curl -fsSL "https://web.archive.org/save/https://iq-me.org/methodology/v1.0.0/en/" > /dev/null
```

The newer snapshot will be returned when visitors access the Wayback Machine
for the canonical URL (IA shows the most recent snapshot by default).

### 4b — Update the snapshot record file

Add a correction entry to
`docs/launch-readiness/internet-archive-snapshots.md`:

```markdown
## Rollback notice for v1.0.0

v1.0.0 was rolled back on [DATE] due to [BRIEF REASON].
The snapshot at [v1.0.0 snapshot URL] reflects the withdrawn version.
A corrected snapshot was captured at [corrected snapshot URL] after redeployment.
```

### 4c — Software Heritage

Software Heritage archives the source repository, not the deployment. The
v1.0.0 source tree will remain in SH permanently (by design). No correction
needed for SH — the git history is tamper-evident and the rollback commits
will be visible in the repository history.

**Dry-run verification:** Confirm `https://web.archive.org/save/` is
accessible. Do NOT call it during dry-run.
